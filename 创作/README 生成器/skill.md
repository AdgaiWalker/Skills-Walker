# README 生成器

基于充分上下文，直接生成或优化 README。

## 执行原则

- 人只负责决策，AI 负责分析、生成、整理与执行
- 不问 AI 自己能判断的问题
- 仅在关键信息缺失时，用少量选项提问
- README 完成后，不立刻结束，要继续给出后续建议

## 适用前提

调用此 skill 时，AI 已有足够上下文，例如：

- 已读取项目代码、文档或已有 README
- 已理解项目用途、受众或用户目标

## 执行流程

1. 基于上下文判断项目类型
2. 直接生成或优化 README
3. 仅在关键信息缺失时再补问
4. README 完成后，给出 LICENSE / Git 后续建议

## README 规则

按项目类型选择最小必要结构，不堆无关章节。

常见类型：

- 库 / 框架
- CLI 工具
- 网站 / 应用
- Skills
- 理论框架
- 文档 / 教程

常见内容：

- 标题 + 一句话介绍
- 安装或使用方法
- 核心亮点或示例
- API / FAQ / 应用场景，仅在需要时加入
- License 说明，仅在已明确或后续需要补充时加入

如项目偏正式发布、品牌展示或社区传播，可优先使用“封面式开头”。

### 封面式开头模板

适用：正式发布、开源项目首页、需要更强展示感的 README。

推荐结构：

    横幅图或品牌图
    H1 标题
    按钮式徽章行
    1 到 2 段高密度介绍

开头规则：

- 优先使用已有横幅图、Logo、截图或品牌素材
- 若项目无现成图片，不虚构外链图片，可退化为标题 + 徽章 + 简介
- 徽章优先放核心跳转，如 Docs、官网、License、社区、发布方
- 开头文案要短，先讲项目是什么，再讲它为什么值得用
- 若采用封面式开头，正文第一屏不要再堆大段背景介绍

## 提问规则

仅在缺少关键决策时，用少量选项提问。

可补问：

- 目标用户：开发者 / 决策者 / 普通用户 / AI 用户 / 思考者
- 风格：极简 / 标准

如上下文已足够，直接生成，不再追问。

## 填充原则

- 如无必要勿增实体
- 保留最小可行示例
- 语言简洁，直击痛点
- 优先保留已有信息，再做重构与美化
- 若许可证尚未明确，不擅自写死具体许可证类型

## 美化规则

已有 README 时：

1. 保留核心信息
2. 调整结构和层级
3. 按所选风格统一视觉层次
4. 补齐必要但缺失的内容

风格：

- 极简：少徽章、少装饰、正文优先
- 标准：适量徽章、清晰分节、表格与引用适度使用
- 标准风格在合适时可使用“封面式开头”

## README 完成后的后续建议

README 完成后，不要直接结束。

AI 先自行检查：

- 是否已有 LICENSE
- README 中是否已有 License 说明
- 当前目录是否为 Git 仓库
- 是否已配置远程仓库
- 是否存在可提交改动

然后一次性给出后续建议，让用户只做决策。

示例表达：

README 已完成。建议继续处理：

- LICENSE：补充或更新许可证
- Git：提交并按需推送本次改动

用户只需决定：

- 两者都处理
- 只处理 LICENSE
- 只处理 Git
- 都不处理

## LICENSE 规则

LICENSE 与 Git 是 README 完成后的两个独立动作，可单独执行，也可同时执行。

若需处理 LICENSE：

- 优先基于上下文推荐，而不是要求用户自行判断
- 常见选项仅保留：
- MIT：宽松，适合大多数项目
- Apache-2.0：宽松，含专利授权
- GPL-3.0：衍生发布需继续开源
- 暂不添加 LICENSE：保留版权，不主动授权

用户确认后：

- 生成或更新 LICENSE 文件
- 同步更新 README 中的 License 说明

## Git 规则

若需处理 Git：

- 检查当前目录是否为 Git 仓库
- 检查是否存在远程仓库
- 检查是否存在可提交改动
- 在用户确认后，再执行提交或推送

## 输出

- 完整流程：README + 后续建议
- 美化 README：只优化 README
- 优化 README：重构内容并美化
- 格式：Markdown，可直接使用
